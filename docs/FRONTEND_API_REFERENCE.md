## 1. Overview & Conventions

| Item | Value |
|---|---|
| Base URL | `http://localhost:5000` |
| API Prefix | `/v1` |
| Auth | Bearer token (`Authorization: Bearer <accessToken>`) |
| Content-Type | `application/json` required for mutating requests |
| Request ID | Auto-attached as `X-Request-Id` header |
| CORS | Configured via `CORS_ORIGIN` env variable |
| Rate Limits | Global: 200 req/min · Auth: 20 req/15min · Analytics: 30 req/min |
| Monetary Values | **All monetary values are in paise** (1 INR = 100 paise). Divide by 100 for display. Send paise in request bodies. |
| Timestamps | ISO 8601 format (`2024-01-15T10:30:00.000Z`) |
| IDs | UUID v4 format |
| Security | Helmet headers, compression, PostgreSQL Row-Level Security per organization |

### Request Headers

```
Authorization: Bearer <accessToken>
Content-Type: application/json
X-Request-Id: <auto-generated-uuid>
```

## 2. Authentication Flow

### Token Lifecycle

| Token | TTL | Storage |
|---|---|---|
| Access Token | 7 days | Memory / localStorage |
| Refresh Token | 30 days | HttpOnly cookie or secure storage |

### JWT Payload Structure

```json
{
  "userId": "uuid",
  "organizationId": "uuid",
  "role": "admin|sales_manager|sales_executive|finance|operations",
  "permissions": ["projects:read", "projects:create", ...],
  "iat": 1700000000,
  "exp": 1700604800
}
```

### Refresh Flow

1. Access token expires → 401 response
2. Call `POST /v1/auth/refresh` with `{ refreshToken }` 
3. Receive new `accessToken` + `refreshToken`
4. **Grace window:** Concurrent refresh requests within a short window return the same new token pair (Redis-backed)
5. Old refresh token is invalidated after successful refresh

### Logout

- `POST /v1/auth/logout` blacklists the access token in Redis
- Blacklisted tokens are rejected on subsequent requests

## 4. All Error Codes

| HTTP | Code | Description |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body failed Zod schema validation. `details` array contains field-level errors. |
| 401 | `AUTHENTICATION_ERROR` | Missing, expired, or blacklisted token. |
| 403 | `AUTHORIZATION_ERROR` | Valid token but insufficient permissions/role. |
| 404 | `NOT_FOUND` | Requested resource does not exist or is not in the caller's org. |
| 409 | `CONFLICT` | Duplicate resource (e.g., duplicate mobile, duplicate booking). |
| 422 | `BUSINESS_RULE_ERROR` | Request violates a business rule (e.g., block limit exceeded, invalid state transition). |
| 429 | `RATE_LIMIT_ERROR` | Too many requests. Retry after the `Retry-After` header value. |
| 500 | `INTERNAL_SERVER_ERROR` | Unexpected server error. Report with `requestId`. |

### Validation Error Details Format

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "statusCode": 400,
    "details": [
      { "path": ["fullName"], "message": "String must contain at least 2 character(s)" },
      { "path": ["mobile"], "message": "Invalid Indian mobile number" }
    ]
  }
}
```

**Lead Status State Machine:**

```
new → contacted → site_visit_scheduled → site_visit_done → interested → negotiation → won
                                                                                     ↘ lost
Any non-terminal state → junk
```

| From | Allowed Transitions |
|---|---|
| `new` | `contacted`, `junk` |
| `contacted` | `site_visit_scheduled`, `interested`, `junk`, `lost` |
| `site_visit_scheduled` | `site_visit_done`, `junk`, `lost` |
| `site_visit_done` | `interested`, `junk`, `lost` |
| `interested` | `negotiation`, `junk`, `lost` |
| `negotiation` | `won`, `lost` |
| `won` | *(terminal)* |
| `lost` | *(terminal)* |
| `junk` | *(terminal)* |

#### `GET /v1/bookings`
**Permission:** `bookings:read`

#### `GET /v1/bookings/:id`
**Permission:** `bookings:read` — Includes payment summary, demand letter count, commission status.

#### `POST /v1/bookings`
**Permission:** `bookings:create`

| Field | Type | Required | Validation |
|---|---|---|---|
| `unitId` | string (UUID) | ✅ | |
| `customerId` | string (UUID) | ✅ | Must have KYC verified |
| `salesPersonId` | string (UUID) | ✅ | Must be active |
| `agentId` | string (UUID) | ❌ | Must be active if provided |
| `tokenAmount` | number | ❌ | In ₹ |
| `discountAmount` | number | ❌ | In ₹ |
| `paymentMode` | enum | ❌ | `cheque` \| `neft` \| `rtgs` \| `upi` \| `dd` \| `cash` |
| `remarks` | string | ❌ | Max 500 chars |

**Pre-flight Validation (all must pass):**
1. Customer must exist and be active with `kycVerified: true`
2. Unit must be in `available`, `blocked`, or `token_received` status
3. If unit is `blocked`, it must be blocked by the same `salesPersonId`
4. Sales person must exist and be active
5. Agent (if provided) must exist and be active

**Discount Rules:**
- ≤ 1% of agreement value → auto-approved

**Cascade Effects (atomic transaction):**
- Unit status → `booked`, linked to customer + booking
- Payment schedule auto-generated from project milestones
- Commission record auto-created if agent present
- Lead (if source lead) → status `won`
- Analytics cache invalidated

## 10. S3 Document Upload Flow

### Two-Phase Upload Process

```
1. Frontend: POST /v1/documents/upload-url
   ← { documentId, uploadUrl, fileKey, expiresInSeconds: 300 }

2. Frontend: PUT <uploadUrl> (direct S3 upload, binary body)
   ← 200 OK from S3

3. Frontend: POST /v1/documents/:documentId/confirm
   ← { documentId, status: "uploaded" }

4. Admin: PATCH /v1/documents/:documentId/verify
   ← { status: "verified" | "rejected" }
```

### Download Flow

```
1. Frontend: GET /v1/documents/:id/download
   ← { downloadUrl, fileName, expiresInSeconds: 3600 }

2. Frontend: GET <downloadUrl> (direct S3 download)
```

> **S3 Key Format:** `{orgId}/{customerId}/{category}/{uuid}_{filename}`  
> **Upload URL expiry:** 5 minutes  
> **Download URL expiry:** 1 hour  
> **Max file size:** 10 MB
